Reddit Tracker — API Call Architecture

How new.json and info.json calls are batched · Rate limit analysis · Capacity planning

1. Overview

Every 60 seconds, the poll worker runs a single global job that covers all active subreddits at once. It makes two types of Reddit API calls per cycle:

  • new.json calls — fetch recently posted content across subreddits
  • info.json calls — refresh engagement scores (upvotes + comments) for all Stack 1 and Stack 2 posts

Reddit's public JSON API allows approximately 60 requests per minute (1 req/sec). Staying under this limit is the core constraint for capacity planning.

Master Formula
Total calls/min = ceil(N / 5) + ceil(N × P / 100)

where  N = number of active subreddits
      P = average number of active Stack 1 + Stack 2 posts per subreddit
Why these numbers? Each new.json batch groups 5 subreddits (CHUNK_SIZE = 5) and fetches 100 posts total (~20/subreddit). Each info.json batch refreshes up to 100 post IDs in a single call. Stack 1 and Stack 2 posts are both refreshed so engagement is current at every transition check.

2. new.json Batching

How it works

Instead of one call per subreddit, all subreddits are grouped into batches of 5 using Reddit's multi-subreddit syntax. Each batch is one HTTP request:

GET /r/sub1+sub2+sub3+sub4+sub5/new.json?limit=100

Reddit returns up to 100 posts from the combined feed of all 5 subreddits — approximately ~20 posts per subreddit. The distribution is based on recency (newest first across all 5), so an unusually active subreddit may take more of the 100 slots.

Call count formula

new.json calls = ceil(N / 5)

Examples:   5 subs → 1 call   10 subs → 2 calls   34 subs → 7 calls  (6 batches of 5 + 1 batch of 4)   50 subs → 10 calls  100 subs → 20 calls

What happens when you add a new subreddit?

No manual action is required. The poll worker queries all active subreddits from the database at the start of every 60-second cycle. A newly added subreddit is automatically included in the next chunked batch — it simply becomes part of the pool. There is no per-subreddit scheduling, no queue entry to create, and no restart needed.

Example: You have 34 subreddits (7 new.json calls). You add 1 more. On the very next poll cycle (within 60 seconds), it runs with 35 subreddits → still 7 calls (ceil(35/5) = 7). No change in call count.

3. info.json Batching

How it works

At each poll cycle, the worker fetches all active Stack 1 and Stack 2 posts from all subreddits in a single database query. These IDs are combined into one flat list and passed to Reddit's global info endpoint in chunks of 100:

GET /api/info.json?id=t3_abc,t3_def,t3_ghi,... (up to 100 IDs per call)

This endpoint accepts post IDs from any subreddit — no subreddit path needed. The response contains the latest upvote and comment counts, which are written back to the database before stack transition checks run.

Which stacks are refreshed, and why?

  • Stack 1 — refreshed ✅. The s1MinEng check happens at transition time (after s1MinAge elapsed), not at insertion time. The post needs current engagement at that moment to be evaluated fairly.
  • Stack 2 — refreshed ✅. Needs live engagement to calculate growth % during the eval window. Without fresh data, growth comparison would use stale values.
  • Stack 3 — not refreshed. Terminal (already alerted). Waiting for 4-hour expiry. No further evaluation needed.

where P = average active Stack 1 + Stack 2 posts per subreddit

Case 2 — Subreddit added mid-cycle

If a subreddit is added while a poll cycle is already running, it is picked up on the next cycle (within 60 seconds). There is no partial-cycle inclusion. The DB query happens at the very start of each job.

Case 3 — Subreddit removed

Removing a subreddit sets isActive = false in the DB. On the next cycle, the worker's query excludes it — no queue cleanup needed. Its existing posts in S1/S2/S3 remain in the DB but will no longer be evaluated or refreshed.

Case 4 — Many active posts (high P)

Watch out: If a subreddit has a long eval window (e.g., s2EvalEnd = 20min) and posts frequently, P can spike. At P=50 with 34 subreddits:
ceil(34/5) + ceil(34×50/100) = 7 + 17 = 24 calls/min — still well within limit.

At P=50, the hard cap is ~85 subreddits before hitting 60 req/min.

Case 5 — One subreddit is dead / private (404)

If a subreddit returns a non-200 response, that chunk is skipped silently (if (!res.ok) continue). Other chunks in the same cycle still run. The subreddit stays in the active list — you would need to manually remove it via the UI.

Case 6 — Rate limited (429)

If Reddit returns 429, the entire poll job throws and BullMQ retries it with exponential backoff (5s → 10s → 20s, up to 3 attempts). This means a rate-limited cycle is skipped rather than partially executed.
